My first non-blog-related project for my apprenticeship was to create a semantic HTML version of a restaurant menu. CSS could be used for simple styles and positioning, but no images could be used. I would use HTML elements to make the code simple, and easy to read and comprehend.

While this project seems simple, it took a lot more thought than I figured it would require.

Teaching myself HTML and CSS was a bit of a trainwreck composed of hacking bits of code together and having it work--but just barely. So I knew that learning to write everything myself from top to bottom would be a real challenge.

For this project, I read Expressive HTML by Billy Whited. I basically started by just writing up a simple HTML document, as I normally would. It was funny to go back and see how the way that I write my HTML mimics the way I would hack different pieces of code together to make sites---disjoined and very ungraceful. I quickly learned that I hardly had any deep understanding of the languages I thought I knew. With the help of my mentor, I was able to make my code more concise and flow a lot better. I feel like I have a more structured and thorough understanding.

I will end my post with a snippet from Billy's post that really summarized my experience:

Expressive html isn’t fundamentally different from “regular” html. It’s not an abstraction. It’s not a new tool. It doesn’t seek to alter the basic vocabulary of the web’s common language.

In fact, even if you aren’t familiar with the term “Expressive html,” I can guarantee you already know how to make use of it.

Rather, Expressive html is simply a different way of seeing, interpreting and authoring markup. It is:

A mindset.
An approach.
A departure.
A mantra.
